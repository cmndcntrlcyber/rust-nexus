# Certificate Pinning and OCSP

## Certificate Pinning Strategy

Nexus uses mTLS with Ed25519 certificates generated by
`scripts/gen-certs-prod.sh`. Pinning is enforced at two levels:

### Server-side (agent authentication)

The server validates incoming client certificates against the CA that
signed them. Only certificates signed by the deployment's CA are
accepted. The CA public key is the effective pin.

### Agent-side (server authentication)

Agents pin the server's CA certificate via the `NEXUS_CA_CERT`
environment variable. This prevents MITM attacks even if a rogue CA
exists in the system trust store.

```bash
NEXUS_CA_CERT=/etc/nexus/ca.crt.pem nexus-agent --config agent.toml
```

### Rotation impact

When certificates are rotated (see `credential-rotation.md`):
1. Generate new certs signed by the same CA — no pin change needed.
2. If the CA itself is re-keyed, all agents must receive the new
   `ca.crt.pem` before the server switches to new certs.

## OCSP Stapling

Nexus does not currently implement OCSP stapling because the mTLS
certificates are self-signed (internal CA, not a public CA).

For deployments using Cloudflare Origin CA certificates:
- Cloudflare handles OCSP for the origin cert at the edge.
- The Nexus server does not need to staple — Cloudflare's edge proxy
  serves the OCSP response to external clients.

For future public-CA integration (e.g., via `instant-acme`):
- OCSP stapling can be enabled in the `rustls` server config.
- Configure `cert_resolver` with a stapled response fetched from the
  CA's OCSP responder on a periodic schedule.
