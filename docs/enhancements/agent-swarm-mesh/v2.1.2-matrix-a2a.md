# MatrixA2A: Design Document for a Hybrid In-Process / gRPC Agent-to-Agent Library with Matrix-Based Routing and Backend-Agnostic Vectorization

## TL;DR

- **Build on the Linux Foundation's A2A protocol as the wire format (not invent your own), implement two transports (in-process Python channel + Tonic-compatible gRPC binding from `specification/grpc/a2a.proto`), and layer a NumPy/Array-API "matrix plane" on top for routing (adjacency, capability, trust matrices) and message transformation (projection, attention).** A2A v0.3+ explicitly defines a gRPC binding alongside JSON-RPC, which is exactly the cross-language Python↔Rust path you need for `rust-nexus` and the other three integration targets.
- **Use a layered architecture**: a transport-agnostic core (`a2a-core`), a swappable transport layer (`a2a-transport-memory`, `a2a-transport-grpc`), a matrix plane (`a2a-matrix` over the Python Array API standard so users can drop in NumPy, PyTorch, JAX, or CuPy), and per-target adapter packages (`a2a-rtpi`, `a2a-btpi`, `a2a-rust-nexus`, `a2a-d3tect`). This keeps the core small and lets each red-/blue-team project pull only what it needs.
- **For the security-sensitive integration targets, default to mTLS over Tonic on Rust side / `grpc.aio.secure_channel` on Python side, capability-based authorization expressed as a matrix `C ∈ {0,1}^(n_agents × n_skills)`, Ed25519-signed AgentCards, and append-only hash-chained audit logs.** Phase delivery in three stages — single-process MVP (weeks 1–4), gRPC interop (weeks 5–10), production hardening + adapters (weeks 11–16).

## 1. Executive Summary

You are designing **MatrixA2A** (working name; pick what you like — `agentmesh`, `a2matrix`, `nexlink` are alternatives), a Python library for agent-to-agent communication with three load-bearing properties:

1. **Hybrid transport.** A single `Channel` abstraction with two concrete implementations: an in-memory channel for sub-microsecond local agent-to-agent messaging, and a gRPC/HTTP-2 channel using the published A2A protobuf for cross-process and cross-language traffic (notably Python ↔ Rust).
2. **Matrix plane.** Two distinct matrix concerns are first-class objects: (a) **topology matrices** (adjacency, capability, trust/permission, reputation, stochastic transition) drive *who can talk to whom and about what*; (b) **transformation matrices** (linear projections, attention QKV, learned routers) shape *how messages are encoded and weighted* between agents.
3. **Backend-agnostic vectorization.** All numerical code uses the [Python Array API standard](https://data-apis.org/array-api/) via `array-api-compat`, so the same routing and similarity code runs on NumPy (default), PyTorch, JAX, CuPy, and Dask without changes — and the library never imposes a particular embedding model.

The design **does not invent a new protocol on the wire**. Google's A2A protocol — donated to the Linux Foundation in June 2025 and now at v0.3 with explicit gRPC binding — already solves agent discovery (AgentCards at `/.well-known/agent-card.json`), task lifecycle, streaming, and security scheme declaration. MatrixA2A is best positioned as a *reference Python implementation of A2A with a matrix routing layer bolted on*, plus adapters for four cybersecurity-focused projects. Inventing new wire semantics will only hurt interop, especially with `rust-nexus` (Rust/Tonic) where the cost of bespoke protobufs is paid every time a `.proto` changes.

The library must be production-grade because two of the integration targets are red-team infrastructure where weak crypto/audit can cause real harm, and two are blue-team detection systems where false negatives have operational cost. Security defaults are non-negotiable: mTLS-only for distributed transport, capability checks before message dispatch, signed AgentCards, hash-chained audit logs.

## 5. Recommended Wire Protocol

**Adopt A2A v0.3+ verbatim.** Pull `specification/grpc/a2a.proto` from `a2aproject/A2A` into `proto/a2a/v1/a2a.proto`. Add one extension file:

```protobuf
// proto/a2a_matrix/v1/matrix.proto
syntax = "proto3";
package a2a_matrix.v1;

import "a2a/v1/a2a.proto";

// Sparse matrix entry for routing/capability/trust matrices.
message MatrixEntry {
  string row_agent_id = 1;
  string col_agent_id = 2;     // or skill_id depending on matrix type
  float  value        = 3;     // adjacency weight, capability bit (0/1), trust score, etc.
}

message Matrix {
  enum Kind {
    ADJACENCY  = 0;
    CAPABILITY = 1;
    TRUST      = 2;
    STOCHASTIC = 3;
    PROJECTION = 4;
  }
  Kind   kind             = 1;
  uint32 rows             = 2;
  uint32 cols             = 3;
  repeated MatrixEntry entries = 4;   // sparse representation
  string version          = 5;        // monotonic version for CAS updates
  bytes  signature        = 6;        // Ed25519 over the canonical encoding
}

service MatrixRouter {
  // Push a new topology to peers (gossip-style).
  rpc PublishMatrix(Matrix) returns (PublishAck);
  rpc SubscribeMatrix(SubscribeRequest) returns (stream Matrix);
  rpc QueryRoute(RouteQuery) returns (RouteResponse);
}

message PublishAck   { string version = 1; }
message SubscribeRequest { Matrix.Kind kind = 1; }
message RouteQuery   { string from_agent = 1; string skill_id = 2; uint32 k = 3; }
message RouteResponse{ repeated string candidate_agent_ids = 1; repeated float scores = 2; }
```

**This is the only thing the Rust side needs to know.** Everything else (NumPy specifics, Array API dispatch, plugin architecture) is a Python implementation detail.

## 7. Core API Design

### 7.1 Defining an Agent

```python
from matrix_a2a import Agent, skill, AgentCard
from matrix_a2a.task import Task, TaskUpdate

class ReconAgent(Agent):
    card = AgentCard(
        name="recon-agent",
        description="Subdomain enumeration and OSINT collection",
        version="1.0.0",
        url="grpc://recon.rtpi.internal:50051",
        capabilities={"streaming": True, "pushNotifications": True},
        default_input_modes=["text/plain", "application/json"],
        default_output_modes=["application/json"],
    )

    @skill(
        id="recon.subdomain_enum",
        name="Subdomain Enumeration",
        description="Enumerate subdomains for a given root domain",
        tags=["recon", "osint", "passive"],
        examples=[{"input": "example.com", "output": ["www.example.com", ...]}],
    )
    async def subdomain_enum(self, task: Task) -> TaskUpdate:
        domain = task.message.parts[0].text
        subdomains = await self._enumerate(domain)
        return TaskUpdate.completed(artifacts=[{"subdomains": subdomains}])
```

### 7.4 Matrix Plane: Topology + Transformation

```python
import numpy as np
from matrix_a2a.matrix import (
    AdjacencyMatrix, CapabilityMatrix, TrustMatrix, AttentionRouter,
)

agents = ["recon", "weaponize", "deliver", "exploit", "c2"]
skills = ["recon.enum", "weapon.payload", "deliver.phish", "exploit.rce", "c2.session"]

# Topology: directed kill chain, no back-edges
A = AdjacencyMatrix(agents, agents, dtype=np.float32)
for src, dst in [("recon","weaponize"),("weaponize","deliver"),
                 ("deliver","exploit"),("exploit","c2")]:
    A[src, dst] = 1.0

# Capability: which agents can perform which skills
C = CapabilityMatrix(agents, skills)
C["recon", "recon.enum"]            = 1
C["weaponize", "weapon.payload"]    = 1
# ...

# Trust: bilinear, decays on failures, increases on successes
T = TrustMatrix(agents, agents, initial=0.5, decay=0.99)

# Compose
router = MatrixRouter(adjacency=A, capability=C, trust=T)
candidates = router.route(from_agent="recon", skill_id="weapon.payload", k=3)
```

### 7.5 Attention-Style Message Transformation

```python
from matrix_a2a.matrix import AttentionRouter

encoder = NumpyEncoder(dim=128)  # any backend; could be HuggingFace, sentence-transformers, BAAI/bge

# Q,K,V projections — pluggable, could be learned

    encoder=encoder,
    n_heads=4,
    head_dim=32,

# Given a query message, route to top-k agents by attention weight

# weights is a softmax-normalized vector; take top-1 or sample stochastically
```

### 7.6 Backend-Agnostic Vector Operations

```python
from matrix_a2a.matrix import array_namespace

def cosine_route(query, agent_embeddings):
    xp = array_namespace(query, agent_embeddings)
    q  = query / xp.linalg.vector_norm(query)
    A  = agent_embeddings / xp.linalg.vector_norm(agent_embeddings, axis=1, keepdims=True)
    return A @ q                  # works for NumPy, PyTorch, JAX, CuPy, Dask
```

This pattern follows the SciPy / scikit-learn approach to Array API support — using `array_api_compat.array_namespace(...)` so the *same code* runs across frameworks. Users supply tensors in their preferred backend and get tensors back in the same backend ("array type in equals array type out").

## 9. Matrix Schemas for Routing

### 9.1 Adjacency Matrix `A ∈ ℝ^(n×n)`
- Rows and columns indexed by agent IDs (or numeric ranks; keep a bijective `id_index` map).
- Element `A[i,j]` is the *capacity* or *willingness* of `i` to send to `j`. `0` means no edge.
- For symmetric topologies, store as upper-triangular sparse; for directed, full sparse.
- Use `scipy.sparse.csr_matrix` or any Array API library's sparse equivalent. Most matrices in agent networks are sparse by nature.
- Eigenvalue properties matter: the Fiedler vector (2nd smallest eigenvalue of the Laplacian) measures connectivity. Provide `A.algebraic_connectivity()` for diagnostics.

### 9.2 Capability Matrix `C ∈ {0,1}^(n_agents × n_skills)`
- Boolean (or fuzzy [0,1] for partial competence).
- Built from union of AgentCard skill declarations.
- Used as a *filter*: candidate agents for skill `s` are `C[:, s].nonzero()`.
- Updated when AgentCards are republished; subscribe via the `MatrixRouter` gRPC service.

### 9.3 Trust / Permission Matrix `T ∈ ℝ^(n×n)`
- Bilinear: `T[i,j]` is `i`'s trust in `j` for receiving its messages.
- Threshold-gated: messages below `θ` are dropped or audited.
- Decays with time: `T ← α·T + (1−α)·T_evidence` after each interaction.
- For red/blue contexts, a *binary* permission matrix is the security baseline; trust scores are reputation overlays.

### 9.4 Stochastic Matrix `P` for Probabilistic Routing
- Row-stochastic: each row sums to 1 (`P 1 = 1`).
- Encodes a Markov-style routing policy: from agent `i`, route to `j` with probability `P[i,j]`.
- Useful for load balancing across redundant agents (e.g., several detection agents in `btpi`).
- Power iteration `Pᵏ` answers "what's the steady-state distribution of messages?"

### 9.5 Projection / Attention Matrices for Transformation
- `W_Q, W_K, W_V ∈ ℝ^(d×d_h)` — standard QKV projections from Transformers.
- Inputs are agent or message embeddings (any encoder); outputs are routing weights via `softmax(QKᵀ/√d_h) V`.
- Optional: per-edge learned weights — `W_ij ∈ ℝ^(d×d)` — for very small agent sets.
- This is essentially a *graph attention* layer applied to the agent topology, exactly the pattern used in MARL communication research (MAGIC, TMAC, MARVIN).

### 9.6 Composability

```python
# Effective routing matrix R = capability ⊙ adjacency ⊙ (trust > θ)
R = C @ A.T  # which skills can be reached from where
R = R * (T > 0.5)  # mask by trust
candidates = R[i, :].nonzero()
```

All composition operators are Array-API-standard so they run on any backend.

## 10. Vectorization API Design

### 10.1 Encoder Protocol

```python
from typing import Protocol, runtime_checkable
import numpy.typing as npt

@runtime_checkable
class Encoder(Protocol):
    """Encode strings or bytes to fixed-dim vectors. Backend-agnostic."""
    dim: int
    def encode(self, x: str | list[str]) -> npt.ArrayLike: ...
    async def aencode(self, x: str | list[str]) -> npt.ArrayLike: ...
```

Concrete implementations live outside the core package as optional extras:
- `matrix_a2a[numpy]` — `RandomProjectionEncoder` (deterministic seeded hash → projection; no model dependency)
- `matrix_a2a[sentence-transformers]` — `SentenceTransformerEncoder`
- `matrix_a2a[torch]` — pluggable PyTorch model
- `matrix_a2a[onnx]` — ONNX Runtime for portable models

The library never *requires* a specific embedding model. Default encoder for "show me it works" is hash-then-project at d=128 with a fixed seed.

### 10.2 Streaming Vectorization

```python
class StreamingEncoder(Encoder):
    """Encodes message chunks as they arrive in a streaming task."""
    async def aencode_stream(
        self, parts: AsyncIterator[Part]
    ) -> AsyncIterator[ArrayLike]: ...
```

Useful in `btpi` for encoding alert streams, or in `d3tect-nexus` for indexing rule documents on the fly.

### 10.3 Vector Store

```python
class VectorStore:
    def __init__(self, *, metric: Literal["cosine","dot","euclidean"] = "cosine"):
        ...
    def add(self, ids: list[str], vecs: ArrayLike) -> None: ...
    def query(self, q: ArrayLike, k: int = 10) -> list[tuple[str, float]]: ...

# Used internally by skill-based routing:
#   skill_descriptions → embedded → stored in VectorStore
#   incoming task message → encoded → top-k matching skills → capability/trust filter
```

For production scale, swap to a managed ANN backend via the same `VectorStore` Protocol (FAISS, Vespa, pgvector, Qdrant). The library ships an in-memory NumPy implementation that's correct and fast up to ~100k agents.

## 12. Security Architecture

### 12.1 Defaults
- **mTLS-only for `GrpcChannel`** — refuse to connect over plaintext in production. Provide an explicit `insecure=True` flag for local dev that warns on every connect.
- **Ed25519-signed AgentCards.** The signature covers the canonical JSON encoding. Verification must happen *before* the card is added to any topology matrix. Without this, the spoofing attack noted in Semgrep's A2A security analysis is trivial.
- **Capability matrix as a hard gate.** Every outbound message goes through `CapabilityCheck.verify(from_agent, to_skill)` before serialization. No bypass.
- **Audit log is append-only and hash-chained.** Each entry has `BLAKE3(prev_hash || record)`. Provides tamper-evidence without requiring a blockchain.

### 12.2 Code Sketch

```python
from matrix_a2a.security import AuditLog, CapabilityCheck, sign_card, verify_card

audit = AuditLog(path="/var/log/matrix-a2a/audit.log")

async def send(from_agent, to_skill, parts):
    if not CapabilityCheck.verify(from_agent, to_skill):
        audit.append({"event": "capability_denied", "from": from_agent, "skill": to_skill})
        raise PermissionError(...)
    audit.append({"event": "send", "from": from_agent, "skill": to_skill, "parts_hash": ...})
    # ...
```

### 12.3 Threat Model Specific to Red/Blue Team Use
- Treat `rtpi` and `rust-nexus` as **handling sensitive offensive capabilities**. Capability matrix should be *closed-by-default* — only explicit grants. Operator authentication via short-lived OIDC tokens piggybacked in gRPC metadata (`authorization: Bearer ...`).
- Treat `btpi` and `d3tect-nexus` as **defensive systems whose disruption matters**. Add rate limits per agent (token bucket in gRPC interceptor). Disable gRPC reflection in production deployments. Cap message sizes (default 4 MB, configurable).
- Per gRPC security best practice, don't put credentials in protobuf message bodies — use channel-level mTLS plus per-call token metadata.

## 14. Testing Strategy

1. **Unit tests** per module, NumPy-only, no network. Keep the matrix algebra deterministic and seed everything.
2. **Property tests with Hypothesis** for matrix invariants: row-stochastic stays row-stochastic under composition, capability filter is monotone, etc.
3. **Integration tests** with both transports: same scenario runs over `InMemoryChannel` and `GrpcChannel`, results must match.
4. **Interop tests Python ↔ Rust**: spin up the Rust crate's example agent in a Docker container; Python test sends a message, asserts response. Run in CI on every PR.
5. **Backend matrix tests** mirroring SciPy's approach: parametrize tests with `xp` fixtures over NumPy, PyTorch (CPU), and JAX. CuPy and Dask in nightly only.
6. **Security regression suite**: confirm capability denial works, signed-card verification rejects tampered cards, audit log detects insertion.

## 15. Phased Roadmap

### Phase 1 — Single-process MVP (4 weeks)
- Goals: prove the API ergonomics; no gRPC, no security hardening.
- Deliverables:

### Phase 2 — gRPC interop (6 weeks)
- Goals: cross-process and cross-language communication via A2A's gRPC binding.

### Phase 3 — Security hardening + adapters (6 weeks)
- Goals: production-ready for the four integration targets.
- Deliverables:
  - Ed25519-signed AgentCards.
  - Hash-chained audit log.
  - Token-bucket rate limits per agent (gRPC interceptor).
  - All four adapters: `a2a-rtpi`, `a2a-rust-nexus`, `a2a-btpi`, `a2a-d3tect`.

- Replace in-mem `VectorStore` with optional FAISS/Qdrant backends.
- Backend support beyond NumPy/PyTorch (JAX/CuPy benchmarked).

- Conformance tests against the official A2A test suite.

### Recommended Triggers / Thresholds
- **If `rust-nexus` already speaks A2A's gRPC binding**, skip Phase 2's adapter wrapper and ship directly against the upstream proto. Confirm before Phase 2 kicks off.
- **If any target uses MCP heavily** (likely `d3tect-nexus`), add an MCP-server-as-skill bridge in Phase 3 — this is small (≤500 LoC) since MCP and A2A are complementary.
- **If you need >10k agents** in a single topology, switch from dense to `scipy.sparse.csr_matrix` and benchmark; the API is unchanged.

## 16. Recommendations (Decision-Ready)

1. **Adopt A2A v0.3+ as your wire protocol.** Inventing a new one will cost weeks and break interop with the wider ecosystem. The Linux Foundation governance and the AWS/Microsoft/Google/Anthropic backing make A2A the safe long-term bet through 2027+.
2. **Use the gRPC binding as your distributed transport, not JSON-RPC.** You explicitly need Python↔Rust interop and bidirectional streaming for agent dialogue; A2A's gRPC binding gives you both natively, while JSON-RPC over SSE adds latency and friction with Tonic.
3. **Treat the matrix plane as orthogonal to the wire format.** Topology and transformation matrices live entirely in the *router*, not in serialized messages (except via the optional `MatrixRouter` extension service). This means future protocol changes don't break the matrix code.
4. **Default to NumPy + Array API standard.** Don't depend on PyTorch — most users won't have it. Make heavy backends optional. This matches SciPy's adoption pattern, which is the most successful precedent.
5. **Lean hard on existing libraries** for cryptography (`cryptography`), gRPC (`grpcio`), validation (`pydantic`), sparse matrices (`scipy.sparse`). Don't re-implement these; the time is better spent on the matrix-routing primitives that don't exist anywhere else.
6. **Get the four target repos in front of you before Phase 3.** The adapter design above is informed inference; some details (specifically `rust-nexus`'s service definitions) will likely need adjustment once the actual code is visible.
7. **Ship a `MatrixA2A SECURITY.md` from day one.** Document your threat model, recommended mTLS topology, audit-log retention policy, and capability-matrix authoring guidance. For red-/blue-team consumers, this is table-stakes.

## 17. Caveats

- **The four cmndcntrlcyber repositories could not be retrieved** during research; the integration-target sections are informed by names plus the operator's adjacent public work (Cobalt Strike profiles, payload tooling, a LangChain pentesting agent system). Validate every adapter-specific claim against the actual repos before implementing.
- **A2A is at v0.3 with v1.0 in active development.** The proto file you vendor will move; pin to a specific A2A release tag, document that pin, and plan a quarterly bump cadence. Some 0.3-era field locations (e.g., `extendedAgentCard`) have been relocated in v1.0 dev to be inside `capabilities`.
- **The Linux Foundation Agentic AI Foundation was formed Dec 2025** taking ownership of A2A *and* MCP. Expect both protocols to consolidate further; design module boundaries so MCP-as-tool can be added without ripping up A2A-as-peer-protocol.
- **gRPC-Rust is in transition.** Google and the Tonic maintainer announced in 2024 that Tonic's channel/server will be replaced with a new implementation under the gRPC org. For the next 12 months `tonic 0.14.x` is correct; after that, plan to track the renamed `grpc-rust` crate. Pin Rust dependencies tightly.
- **Performance claims about matrix routing depend on sparsity.** A 10k-agent fully-connected adjacency is 800 MB at fp64. In practice, agent networks are sparse (degree typically <100 per node), so use sparse representations from the start. Benchmark before claiming scalability numbers.
- **Some of the agent-framework comparison data above is forward-looking** — sources like Intuz, Gurusup, and OpenAgents speculate on 2026 trends and dollar costs that I did not independently verify. Treat framework comparisons as orientation, not procurement guidance.
- **Cosine similarity for capability matching has known frequency-bias issues** (Steck et al., Netflix 2024) with learned embeddings. For high-precision routing, consider unnormalized dot products or learned routers rather than blindly using cosine; provide both as options in `VectorStore.metric`.
- **mTLS is necessary but not sufficient.** Long-lived bidi streams in gRPC need periodic re-authorization checks for changing roles/scopes; bake that into the streaming interceptor in Phase 3.
- **Don't ship reflection in production.** Disable `tonic-reflection` and `grpc-reflection` for the gRPC channel in deployments; they trivially map your API surface for an attacker.
