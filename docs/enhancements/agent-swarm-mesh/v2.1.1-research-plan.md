# v2.1.1 — GML Noise Barometer: Research Plan

v2.1.1 is **research-phase**. The source doc ([`v2.1.1-gml-agent-swarm-telemetry-analysis.md`](v2.1.1-gml-agent-swarm-telemetry-analysis.md)) is a 22-line task list, not an implementation plan. This file structures the research **output** into milestones; the deliverable of this research is a new design doc (`v2.1.1-gml-implementation-design.md`) that will in turn seed a future `v2.1.1-implementation-plan.md`.

Tracked in [`STATUS.md`](STATUS.md).

---

## Goal

Produce a written design for a Graph-Machine-Learning pipeline that:

1. Models agents/hosts/services as graph nodes; network flows and inter-agent communications as edges.
2. Ingests temporal graph snapshots (hourly) of message volume and telemetry-event intensity from `nexus-mesh` (Gossipsub).
3. Outputs an anomaly score + alert prediction — a "noise barometer."
4. Drives a hysteresis-based control loop that throttles per-agent activity rates when SIEM alert pressure rises.
5. Includes safeguards against label leakage, adversarial robustness issues, and model drift.

Per the source doc's roadmap hint: **start with heterogeneous GraphSAGE on hourly snapshots, supervised on SIEM alert counts; run in shadow mode for validation; then enable throttling with conservative rate floors.**

## Research milestones

### R1 — Architecture survey

**Goal.** Survey graph-ML architectures and produce a 1-page tradeoff matrix.

**Sources.**
- Hugging Face graph-ml model index: `https://huggingface.co/models?pipeline_tag=graph-ml&sort=likes`
- Singularity model card: `https://huggingface.co/SlappAI/Singularity` (for RGNN / Dynamic Relationship Expansion / AGNN concepts)
- Standard architectures: GCN, GraphSAGE, GAT, R-GCN, Graph Transformers

**Tradeoffs to document.**
- Aggregation function choice (mean / max / sum / attention).
- Depth vs over-smoothing.
- Sampling strategies (neighbor sampling, GraphSAINT-style).
- Inductive vs transductive — **must be inductive** for our use case since the swarm topology changes when agents come and go.
- Heterogeneous edge types (agent↔server is different from agent↔agent).

**Output.** A page in the future design doc with a comparison table and a recommendation. Hint from source: heterogeneous GraphSAGE is the recommended starting point.

**Exit criterion.** Tradeoff matrix written; architecture recommendation made (resolves D-2.1.1-A).

**Estimated size.** 1–2 days.

---

### R2 — Noise-barometer math

**Goal.** Define how mesh telemetry + SIEM alerts become training data and model output.

**Sub-questions.**
- **Snapshot construction.** Hourly windows. For each window, build a heterogeneous graph: node features (per-agent message volume, per-host service count, per-edge byte-rate); edge types (`agent↔server`, `agent↔agent`, `host↔service`).
- **Labels.** Map SIEM alert counts per window per host → graph-level or node-level label. Decide: regression on count vs classification on threshold-crossing.
- **Anomaly-scoring function.** Output of the trained model + comparison to a rolling baseline. Define the math explicitly (z-score over a 7-day rolling mean? probabilistic — predicted alert count above 95th percentile?).
- **Telemetry pipeline.** How Gossipsub messages aggregate into per-window counters. Cheap counting structures (`HashMap<(peer, kind), u64>`) flushed at window boundary.

**Output.** A "Data and Model" section in the design doc. Includes a worked example on synthetic data (since v2.1 Phase 4 may not have landed when R2 starts).

**Exit criterion.** Math defined; synthetic-data prototype runs end-to-end and produces a scalar barometer value per window. SIEM source named (resolves D-2.1.1-B).

**Estimated size.** 3–5 days; longer if v2.1 Phase 4 is in place and real telemetry is incorporated.

---

### R3 — Control-loop design

**Goal.** Specify the closed-loop controller that consumes the barometer and adjusts per-agent activity rates.

**Sub-questions.**
- **Hysteresis thresholds.** Two thresholds (engage / disengage), not one, to prevent oscillation. Define both in terms of the barometer's scale from R2.
- **Conservative rate floors.** Minimum activity rates per agent — never throttle to zero silently. Per the source doc: "never deploy silent activity throttling without operator visibility."
- **Per-agent attribution.** Use edge-attention weights from the R1 model (works naturally for GAT and attention-based variants; for GraphSAGE, layer a small attention head on top of the embeddings).
- **Recovery dynamics.** How quickly rates restore once alert pressure subsides. Time constants in minutes, not hours, to avoid extended throttle "stuck" states.

**Output.** A "Control Loop" section in the design doc, including a state diagram and pseudocode.

**Exit criterion.** Pseudocode written; threshold rationale documented; per-agent attribution mechanism named.

**Estimated size.** 2–3 days. Can run in parallel with R2.

---

### R4 — Safeguard analysis

**Goal.** Enumerate failure modes and mitigations.

**Failure modes to cover.**
- **Label leakage.** Are SIEM alerts caused by an agent that *received* a task in the same window? That's a self-fulfilling loop. Mitigation: lag the label by N windows or exclude same-window labels for the originating agent.
- **Adversarial robustness.** A blue-team analyst who knows the model exists may craft inputs to silence agents. Mitigation: don't expose barometer output on agent-side network surface; restrict to operator dashboard.
- **Model drift.** Operator changes SIEM rules → label distribution shifts → model goes stale silently. Mitigation: periodic re-training schedule + drift-detection metric (PSI on input features, KS test on output distribution).
- **Failure mode of the throttle itself.** What if the controller is what's wrong — a bug throttles all agents to floor. Mitigation: kill switch (parallel to the v2.1 Phase 10 mesh kill switch) that disables throttling and reverts to baseline rates. Signed by offline operator key.

**Output.** A "Safeguards" section in the design doc.

**Exit criterion.** All four failure modes have documented mitigations; kill-switch design specified.

**Estimated size.** 2 days.

---

### R5 — Roadmap synthesis

**Goal.** Produce the complete `v2.1.1-gml-implementation-design.md` design doc that integrates R1–R4 outputs, and a follow-up implementation plan.

**Output.**
1. `v2.1.1-gml-implementation-design.md` — the complete design (a sibling of `v2.1.2-matrix-a2a.md` in shape and depth).
2. A skeleton `v2.1.1-implementation-plan.md` outlining post-research phases:
   - **Implementation Phase 1 — Shadow mode (M5).** Train on logged telemetry + SIEM, predict but don't throttle. Surface predictions on `nexus-webui`. Resolves D-2.1.1-C.
   - **Implementation Phase 2 — Throttle go-live (M6).** Conservative rate floors; full kill switch; one engagement of production use.
   - **Implementation Phase 3 — Retraining + drift detection.** Scheduled retraining, drift alarms, kill-switch end-to-end test.

**Exit criterion.** Both docs exist; the design doc passes a documentation-review pass from the v2.1 + v2.1.2 owners (the GML controller needs to address agents via mechanisms those tracks provide).

**Estimated size.** 2–3 days for the design doc itself; the skeleton implementation plan is 1 day.
