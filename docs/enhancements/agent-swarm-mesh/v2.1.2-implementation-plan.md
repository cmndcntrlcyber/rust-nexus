# v2.1.2 — MatrixA2A Interop: Implementation Plan

Actionable, tracked plan for the v2.1.2 swarm-mesh track. The full design (large, Python-centric) lives in [`v2.1.2-matrix-a2a.md`](v2.1.2-matrix-a2a.md). This file extracts only the **rust-nexus-facing work** and re-casts it into execution-ready phases tracked against [`STATUS.md`](STATUS.md).

---

## Pre-flight decision (Phase 0)

Three decisions must be resolved in [`ROADMAP.md`](ROADMAP.md) before Phase 1 opens:

- **D-2.1.2-A — Hosting model for `a2a-matrix-rs`.**
  - **Recommended: host as workspace crate.** Add a new `nexus-a2a/` member to the rust-nexus `Cargo.toml` `members` list. Vendor `a2a.proto` and `matrix.proto` under `nexus-a2a/proto/`. Matches existing crate naming and lets `tonic-build` run as part of the workspace build, parallel to how `nexus-infra` builds `nexus.proto`.
  - *Alternative: external crate.* `a2a-matrix-rs` lives in a separate repo, consumed via crates.io or git dep. Introduces release-coupling friction and a second CI surface; not recommended.
- **D-2.1.2-B — A2A version pin.** Pin to a specific upstream tag from `a2aproject/A2A` (e.g., `v0.3.1`). Plan a quarterly cadence bump. As of source-doc writing, v0.3 was stable with v1.0 in development; some 0.3-era field locations relocate in v1.0 dev (e.g., `extendedAgentCard` moves inside `capabilities`).
- **D-2.1.2-C — Proto-surface separation from `nexus.proto`.** Both protos coexist in the workspace; **no method overlaps**. `nexus.proto` stays for C2 internals (agent registration, task streaming, file transfer). `a2a.proto` is the public interop surface (skills, A2A tasks, AgentCards). Documented and code-reviewed at Phase 1.

Also resolve **D-XLINK-A** (mesh ↔ A2A transport semantics) in [`ROADMAP.md`](ROADMAP.md). Recommended default: A2A over Tonic gRPC primary; A2A may optionally tunnel over `nexus-mesh::publish` once v2.1 Phase 7 is in place.

## Scope cuts

Explicit out-of-scope items pulled from the source design doc — these are Python-side or external concerns that **do not** land in rust-nexus:

- The Python library `matrix-a2a/src/matrix_a2a/` and its modules (`channel`, `matrix`, `router`, `vector`, `security`, `adapters`).
- Python adapters: `a2a-rtpi`, `a2a-btpi`, `a2a-d3tect`. (`a2a-rust-nexus` Python adapter is a *consumer* of the Rust server; we ship the server, not the Python wrapper.)
- Array-API backends (NumPy, PyTorch, JAX, CuPy).
- VectorStore / encoder ecosystem.
- The Python documentation site.

What rust-nexus ships: the `.proto` files, the Tonic build, an A2A server implementation, an A2A client, mTLS plumbing, signed AgentCards, the audit log, the capability matrix (as a Rust struct using `nalgebra` or `ndarray`), and the matrix router service.

## Phase 1 — Vendor protos + Tonic build

**Goal.** A new `nexus-a2a/` workspace crate compiles, generates Rust code from `a2a.proto` and `matrix.proto`. No service implementations yet.

**Files touched.**
- `Cargo.toml` (add `"nexus-a2a"` to workspace `members`)
- `nexus-a2a/Cargo.toml` — `tonic`, `prost`, `prost-types`, `tokio`, `tokio-stream`, `nalgebra` *or* `ndarray`; `[build-dependencies] tonic-build`
- `nexus-a2a/build.rs` — vendor-compile both protos
- `nexus-a2a/proto/a2a/v1/a2a.proto` (vendored from `a2aproject/A2A` at the pinned tag)
- `nexus-a2a/proto/a2a_matrix/v1/matrix.proto` (new, defined in source doc §8)
- `nexus-a2a/src/lib.rs` — re-exports

**Deliverables.**
- The two vendored protos.
- `build.rs` configured per source doc:
  ```rust
  tonic_build::configure()
      .build_server(true)
      .build_client(true)
      .compile_protos(
          &["proto/a2a/v1/a2a.proto", "proto/a2a_matrix/v1/matrix.proto"],
          &["proto"],
      )?;
  ```
- A `LICENSE-A2A` file capturing the upstream A2A license (Apache-2.0).
- A `VENDORED-VERSION` file naming the upstream A2A tag pinned (resolves D-2.1.2-B in writing).

**Verification.** `cargo build -p nexus-a2a` succeeds; generated types are visible via `cargo doc -p nexus-a2a --no-deps`.

**Exit criteria.** Workspace builds clean; `nexus-a2a` doc-build shows `AgentCard`, `Task`, `Message`, `A2aService`, `MatrixRouter`, `Matrix` types.

**Dependencies.** D-2.1.2-A, D-2.1.2-B, D-2.1.2-C resolved.

**Rollback.** Remove `nexus-a2a` from workspace `members`; the crate directory can stay isolated.

**Estimated size.** ~150 LoC (Cargo.toml + build.rs + vendored .proto + re-exports), ~1 day.

## Phase 2 — A2AService server + client with mTLS

**Goal.** A rust-nexus A2A server and a rust-nexus A2A client can complete a streaming task. A Python A2A client using the same proto interoperates. **Milestone M2.**

**Files touched.**
- `nexus-a2a/src/server.rs`, `nexus-a2a/src/client.rs`
- `nexus-a2a/src/mtls.rs`
- `nexus-a2a/examples/echo_server.rs`, `nexus-a2a/examples/echo_client.rs`

**Deliverables.**
- `NexusAgent` struct implementing `A2aService`. The 11 RPCs per A2A v0.3 — `SendMessage`, `SendStreamingMessage`, `GetTask`, `CancelTask`, `TaskSubscription`, `CreateTaskPushNotificationConfig`, `GetAgentCard`, `ListTask`, plus the remaining three. Each method delegates to a pluggable handler trait.
- `A2aClient` wrapping a Tonic channel with mTLS configuration.
- `MtlsConfig::from_files(ca, cert, key)` and `MtlsConfig::from_env()` (env var names `NEXUS_CA_CERT`, `NEXUS_CLIENT_CERT`, `NEXUS_CLIENT_KEY` per source doc convention).
- Echo example: server returns an artifact echoing the message; client sends `"hello"`, prints reply.
- A Python interop fixture: `tests/interop/python_client.py` using `grpcio` against the same vendored proto. Run in CI inside a small Docker container.

**Verification.** `cargo run --example echo_server -p nexus-a2a` + `cargo run --example echo_client -p nexus-a2a` complete a round-trip. Python interop test passes in CI.

**Exit criteria.** Both examples run; Python ↔ Rust interop test green in CI. **Milestone M2 reached.**

**Dependencies.** Phase 1.

**Rollback.** Crate-local revert. No callers outside `nexus-a2a` yet.

**Estimated size.** ~600 LoC + Python fixture, ~4 days.

## Phase 3 — Security hardening

**Goal.** Production-ready security defaults. **Milestone M4.**

**Files touched.**
- `nexus-a2a/src/security/mod.rs`, `nexus-a2a/src/security/cards.rs`, `nexus-a2a/src/security/audit.rs`, `nexus-a2a/src/security/caps.rs`, `nexus-a2a/src/security/rate_limit.rs`

**Deliverables.**
- **Ed25519-signed AgentCards.** Signature covers the canonical JSON encoding. Verification happens *before* a card is added to any topology matrix. Reuse `NodeIdentity` from v2.1 Phase 1 — same Ed25519 key serves both mesh and A2A identity.
- **Capability matrix as a hard gate.** Every outbound `SendMessage` runs through `CapabilityCheck::verify(from_agent, to_skill)` before serialization. No bypass path.
- **Hash-chained audit log.** Append-only file. Each record: `BLAKE3(prev_hash || record_bytes)`. Per source doc — tamper-evident, no blockchain. Pluggable backend (file, syslog) via a trait.
- **Token-bucket rate limit** as a Tonic gRPC interceptor. Default: 100 RPS per agent, configurable.
- **mTLS-only refusal** in production builds. Provide `insecure=true` flag for local dev that warns on every connect.
- **gRPC reflection disabled** in production (don't ship `tonic-reflection` for the A2A service in release builds — trivially maps the API surface for an attacker).
- **Message size cap** default 4 MB, configurable.

**Verification.** Security regression suite — confirm capability denial works, signed-card verification rejects tampered cards, audit-log insertion detection works, rate-limit interceptor drops over-budget requests.

**Exit criteria.** All security tests pass; production-mode plaintext connection refused with a clear error. **Milestone M4 reached.**

**Dependencies.** Phase 2; v2.1 Phase 1 (`NodeIdentity`) if reusing.

**Rollback.** Per-feature config flags; full rollback via `git revert` of security module commits.

**Estimated size.** ~700 LoC + security tests, ~5 days.

## Phase 4 — Matrix router + interop tests

**Goal.** The `MatrixRouter` extension service is live; routing decisions consult adjacency × capability × trust matrices.

**Files touched.**
- `nexus-a2a/src/matrix/mod.rs`, `nexus-a2a/src/matrix/adjacency.rs`, `nexus-a2a/src/matrix/capability.rs`, `nexus-a2a/src/matrix/trust.rs`, `nexus-a2a/src/matrix/stochastic.rs`
- `nexus-a2a/src/router.rs`

**Deliverables.**
- `AdjacencyMatrix`, `CapabilityMatrix`, `TrustMatrix`, `StochasticMatrix` over `nalgebra` *or* `ndarray` (decided at Phase 1). Sparse representations where appropriate (the source doc warns: a 10k-agent fully-connected adjacency is 800 MB at fp64).
- `MatrixRouter` service implementing `PublishMatrix`, `SubscribeMatrix`, `QueryRoute` from `matrix.proto`.
- Routing composition per source doc §9.6: `R = capability ⊙ adjacency ⊙ (trust > θ)`.
- Reputation decay on `TrustMatrix`: `T ← α·T + (1−α)·T_evidence` after each interaction.
- Property tests on matrix invariants (Hypothesis-style; use `proptest`): row-stochastic stays row-stochastic under composition, capability filter is monotone, signed-matrix versions verify.
- An interop test that runs the same scenario over a Python `InMemoryChannel` (from the upstream `matrix-a2a` Python lib, if present in CI) and the Rust `GrpcChannel` — results must match.

**Verification.** `cargo test -p nexus-a2a matrix::` and `cargo test -p nexus-a2a router::` pass; property tests pass; cross-language interop test passes.

**Exit criteria.** All tests green; example showing matrix-routed dispatch runs.

**Dependencies.** Phase 3.

**Rollback.** Crate-local.

**Estimated size.** ~800 LoC + property tests, ~5 days.

## Mesh interop checkpoint (post-Phase 4)

Once v2.1 Phase 7 (mesh wired into `nexus-web-comms`) and v2.1.2 Phase 4 are both live, evaluate whether A2A `Message` envelopes should also be carriable over `nexus-mesh::publish` as a sealed payload. Per D-XLINK-A default:

- A2A gRPC remains the primary control plane for structured agent dialogue.
- libp2p mesh remains the fallback transport when gRPC is blocked.
- Tunneling A2A over mesh is an optional Phase 5 (~3 days, ~300 LoC) — adds an `A2aOverMesh` adapter to `nexus-web-comms`. Defer unless an engagement requires it.

## Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| A2A proto bumps mid-implementation (v0.3 → v1.0 churn during dev window) | medium | medium — rework | Pin to specific upstream tag at Phase 1 (D-2.1.2-B); document the pin; bump only between phases |
| Tonic/grpc-rust transition during dev window | medium | medium — rebase | Pin `tonic = "0.14"` per source doc; revisit only after track ships |
| `nexus.proto` and `a2a.proto` accidentally develop overlapping methods | low | high — operator confusion | Phase 1 code review explicitly checks for method-name collision; documented in D-2.1.2-C |
| Signed AgentCard verification skipped on a fast path | medium | very high — spoofing | Code-review gate at Phase 3; integration test specifically attempts to register a tampered card and asserts rejection |
| Audit log loses ordering under crash mid-write | low | high — tamper-evidence weakened | Use `fsync`-after-write append model; documented; recovery procedure in operator docs |
| Cosine-similarity frequency bias for skill matching (Steck et al., Netflix 2024) | medium | medium — wrong routing | Phase 4 exposes both `cosine` and `dot` metrics on `VectorStore` (Rust side); document the bias |
| Long-lived bidi streams skip re-authorization as roles change | medium | high — stale capability | Phase 3 streaming interceptor performs periodic re-auth check (default every 5 minutes) |
| Adapter scope creep into rust-nexus | high if not policed | medium — bloat | Explicit scope-cut list at top of this file; PR template question: "is this a Python concern?" |
