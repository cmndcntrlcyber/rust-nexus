## Phase 1 â€” Ed25519 identity in `nexus-common`

**Goal.** Every node has a stable cryptographic identity (Ed25519 signing + X25519 KEM) independent of TLS certs and `agent_id`.

**Files touched.**
- `nexus-common/Cargo.toml` (add `ed25519-dalek`, `x25519-dalek`)
- `nexus-common/src/identity.rs` (new)
- `nexus-common/src/crypto.rs`
- `nexus-common/src/lib.rs` (re-export `identity` module)

**Deliverables.**
- `NodeIdentity { ed25519_signing: SigningKey, x25519_secret: StaticSecret }` with `generate()`, `from_seed(&[u8; 32])`, `peer_id() -> [u8; 32]` (BLAKE3 of ed25519 public key), `sign(&[u8])`, `verify(...)`.
- Disk persistence (path from config), load-on-startup, generate-and-save on first run, cross-platform (Windows + Linux).
- Unit tests: round-trip persist/load, sign/verify, peer-id determinism.

**Verification.** `cargo test -p nexus-common identity::` â€” all tests pass.

**Exit criteria.** Existing agent + server build and run unchanged (nothing calls `NodeIdentity` yet); new tests pass.

**Dependencies.** Phase 0; D-2.1-B resolved.

**Rollback.** `git revert` is safe; nobody else depends on the new module yet.

**Estimated size.** ~250 LoC, ~1 day.

## Phase 5 â€” End-to-end encryption above Gossipsub (`SealedEnvelope`)

**Goal.** A `Message` published to peer X's inbox topic is opaque to every other peer on the mesh, even though Gossipsub broadcasts the topic publicly.

**Files touched.**
- `nexus-mesh/src/sealed.rs` (new), `nexus-mesh/src/pubsub.rs`

**Deliverables.**
- `SealedEnvelope { sender_pubkey, ephemeral_x25519_pub, nonce, ciphertext, signature }`.
- Sender flow: ephemeral X25519 keypair â†’ ECDH with recipient's static X25519 â†’ HKDF â†’ AES-256-GCM key â†’ encrypt bincoded `Message` â†’ sign envelope with sender's Ed25519.
- Recipient flow: verify signature â†’ ECDH â†’ decrypt. Reject any failure.
- Public-key directory populated via Identify extension or signed announce on `nexus/keys`.
- Replay protection: per-sender sliding-window nonce cache, drop on duplicate.

**Verification.** 3-node example: A â†’ C sealed message; B in the middle logs the Gossipsub frame and confirms it cannot decrypt. Fuzz test for malformed envelopes.

**Exit criteria.** Example demonstrates plaintext opacity at the relay peer; fuzz test runs without panic.

**Dependencies.** Phase 4.

**Rollback.** Crate-local.

**Estimated size.** ~400 LoC, ~2â€“3 days.

## Phase 6 â€” Store-and-forward DTN queue

**Goal.** A sender can publish to a recipient that's offline; recipient receives on reconnect.

**Files touched.**
- `nexus-mesh/Cargo.toml` (add `redb`)
- `nexus-mesh/src/dtn.rs` (new)

**Deliverables.**
- `redb`-backed outbound queue: when `SealedEnvelope` is published and Gossipsub reports zero subscribed peers for the recipient inbox, persist locally with a TTL.
- Drain on peer-connected events for queued recipients.
- Per-peer queue cap with newest-first rejection on full (anti-amplification).
- Receiver-side dedupe by envelope hash.

**Verification.** Integration test: start A and C, A sends to C, both disconnect, A reconnects, C reconnects, C sees the message exactly once.

**Exit criteria.** Integration test passes; per-peer queue cap enforced under load test.

**Dependencies.** Phase 5.

**Rollback.** Crate-local. Note: `redb` on-disk files in the configured DTN directory should be deleted on rollback to avoid stale-format issues if Phase 6 is re-attempted later.

**Estimated size.** ~350 LoC, ~2 days.

**Deliverables.**
- Extract a `Transport` trait inside `nexus-web-comms` (currently `WebCommsClient` has methods, no trait).

- Config `[mesh] enabled=false, bootstrap=[â€¦], identity_path="â€¦"` (off by default).
- Fallback ordering when enabled: gRPC â†’ mesh â†’ HTTP â†’ WebSocket. Rationale: gRPC is cheapest when reachable; mesh routes through peers exactly when gRPC has been blocked.

## Phase 8 â€” Multi-hop relay and routing policy

**Goal.** A deeply-sandboxed agent reaches the C2 via two or more hops. **Milestone M3.**

**Files touched.**
- `nexus-mesh/src/sealed.rs`, `nexus-mesh/src/pubsub.rs`, `nexus-mesh/src/relay.rs` (new)

**Deliverables.**
- Per-message hop counter + TTL on `SealedEnvelope`; decrement on rebroadcast; drop at 0 (loop prevention).
- Explicit "relay-only" peer mode â€” accepts and rebroadcasts for topics it isn't subscribed to. Used for operator-owned staging boxes.
- Metrics: messages delivered, dropped (TTL/queue-full/dedupe), median hops to server. Surface via existing logging.
- Optional: deny-list of suspected-burned peer IDs that this node refuses to relay for.

**Verification.** 5-node line topology (A â€” B â€” C â€” D â€” server); A's heartbeat reaches the server with hop=3; tasks flow back the same way.

**Exit criteria.** 5-node E2E test passes; metrics surface in logs.

**Dependencies.** Phase 7.

**Rollback.** Disable relay-only mode in config.

**Estimated size.** ~250 LoC, ~1 day.

## Phase 9 â€” Sphinx mix-style sender anonymity (OPTIONAL)

**Goal.** An attacker who owns one mesh peer cannot tell which agent originated a given message. **Only adopt if the threat model in Phase 0 demands it.** Adds latency and complexity.

**Pre-flight.** Resolve D-2.1-C: per-file license verification of `ghostwire-libs/sphinx-rs` (MIT/Apache, separate from AGPL parent) OR pick `sphinx-packet` alternative.

**Files touched.**
- `nexus-mesh/src/sphinx.rs` (new), `nexus-mesh/Cargo.toml`

**Deliverables.**
- Fixed-length onion packet wrapping `SealedEnvelope`, with N intermediate hops chosen from the Kademlia routing table.
- Cover traffic: low-rate scheduled decoy message per node, same Sphinx packet length so observers can't distinguish real sends from cover.
- Per-operation opt-in config flag; not always-on.

**Verification.** Informal traffic-analysis test: an observer sees N nodes' egress and cannot correlate origin above the rate of cover traffic. (Formal anonymity audit out of scope.)

**Exit criteria.** Test passes; latency cost documented in operator config.

**Dependencies.** Phase 8; D-2.1-C resolved.

**Rollback.** Per-operation flag off; code is feature-gated.

**Estimated size.** ~600 LoC, ~3â€“4 days. **Skip unless a specific engagement requires it.**

**Exit criteria.** Operator dry-run completes; kill switch tested end-to-end.

**Dependencies.** Phase 8 (Phase 9 only if enabled).

**Rollback.** Phased feature-flag flip; full opt-out via config.

**Estimated size.** ~200 LoC + docs, ~2 days.

## Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| libp2p major version churn during the 16-day implementation window | medium | medium â€” rebase cost | Pin to `0.53` at Phase 2 start; revisit only between phases, never mid-phase |
| `sphinx-rs` per-file license inconsistency vs the AGPL parent project | low | high â€” license contamination | **Mandatory** per-file header verification before vendoring (Phase 9 gate); if any file lacks a clear MIT/Apache header, swap to `sphinx-packet` |
| DTN queue disk amplification under hostile peers | medium | medium â€” operator disk exhaustion | Phase 6 per-peer cap with newest-first rejection; documented operator-visible queue size metric |
