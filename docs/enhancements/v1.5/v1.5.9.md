# Part I — Multiple Origin Certificate Paths

The infrastructure currently supports a single origin certificate across the
entire system. `OriginCertConfig` holds one `cert_path`/`key_path`/`ca_cert_path`,
`CertManager` loads one cert chain, and `GrpcServer` binds with a single TLS
identity. This blocks multi-domain deployments (primary C2 + fallback domain),
domain fronting with distinct TLS identities, and concurrent listeners with
different certificates.

Part I introduces named HTTPS profiles — each with its own origin cert, key,
and CA — selectable by SNI hostname or explicit profile name.

```rust
// Validate HTTPS profiles (if present)
if let Some(profiles) = &self.origin_cert.profiles {
    let mut seen_domains: HashMap<&str, &str> = HashMap::new();
    for profile in profiles {
        if profile.name.is_empty() {
            return Err(InfraError::ConfigError(
                "HTTPS profile name cannot be empty".to_string(),
            ));

                return Err(InfraError::ConfigError(format!(

                    domain, existing, profile.name,
                )));
            }
            seen_domains.insert(domain, &profile.name);
        }
    }
}
```

```rust
/// Check whether any named profiles are configured
pub fn has_profiles(&self) -> bool {
    !self.profiles.is_empty()
}

/// Look up a profile by SNI hostname (exact match, then wildcard fallback)
pub fn resolve_profile(&self, sni: &str) -> Option<&CertProfile> {
    if let Some(name) = self.domain_index.get(sni) {
        return self.profiles.get(name);
    }
    // Wildcard fallback: strip first label, check *.rest
    if let Some(dot) = sni.find('.') {
        let wildcard = format!("*{}", &sni[dot..]);
        if let Some(name) = self.domain_index.get(&wildcard) {
            return self.profiles.get(name);
        }
    }
    None
}

/// Get a profile by name
pub fn get_profile(&self, name: &str) -> Option<&CertProfile> {
    self.profiles.get(name)
}

/// List all profile names
pub fn profile_names(&self) -> Vec<&str> {
    self.profiles.keys().map(|s| s.as_str()).collect()
}
```

## Stage 9.3 — SNI-based cert resolution for `GrpcServer`

**Scope:** Implement `rustls::server::ResolvesServerCert` for multi-profile
SNI dispatch. Integrate with `GrpcServer::start()`. Existing single-cert
path unchanged when no profiles are configured.

**Step 9.3.1** — Add `MultiProfileCertResolver` to `cert_manager.rs`:

```rust
use rustls::server::{ClientHello, ResolvesServerCert};
use rustls::sign::{any_supported_type, CertifiedKey};

pub struct MultiProfileCertResolver {
    /// Profile name -> pre-built CertifiedKey
    profiles: HashMap<String, Arc<CertifiedKey>>,
    /// SNI domain -> profile name
    domain_index: HashMap<String, String>,
    /// Fallback when no SNI match
    default: Arc<CertifiedKey>,
}

impl ResolvesServerCert for MultiProfileCertResolver {
    fn resolve(&self, client_hello: ClientHello<'_>) -> Option<Arc<CertifiedKey>> {
        let sni = client_hello.server_name()?;
        if let Some(name) = self.domain_index.get(sni) {
            self.profiles.get(name).cloned()
        } else {
            Some(Arc::clone(&self.default))
        }
    }
}
```

**Step 9.3.2** — Add `CertManager::create_multi_profile_server_config()`:

Builds a `rustls::ServerConfig` using `with_cert_resolver(Arc::new(resolver))`
instead of `with_single_cert()`. Wraps each profile's `PrivateKey` with
`any_supported_type()` to produce a `SigningKey`. The default profile is
built from the legacy single-cert fields.

When `mutual_tls` is true, the `ClientCertVerifier` is constructed from the
union of all profiles' CA certificates.

Known error categories:

| Error | Root Cause | Fix |
