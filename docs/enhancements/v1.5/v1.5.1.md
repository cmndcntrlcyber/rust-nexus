## Stage 1.1 ‚Äî Fix compile-blocking tonic import

`nexus-agent/Cargo.toml:31` aliases tonic 0.14 as `tonic_14`, but
`nexus-agent/src/a2a_client.rs:45` uses bare `tonic::transport::ClientTlsConfig`.

**Compiler error:**
```
error[E0433]: cannot find module or crate `tonic` in this scope
  --> nexus-agent/src/a2a_client.rs:45:27
   |
45 | fn client_tls() -> Option<tonic::transport::ClientTlsConfig> {
   |                           ^^^^^ use of unresolved module or unlinked crate `tonic`
```

**Scope:** Exhaustive grep confirms line 45 is the ONLY bare `tonic::` reference
in `nexus-agent/src/`. All other tonic usage in the crate goes through the
`nexus_a2a` re-exports (`A2aClient`, `tls`, `pb`) which handle the namespace
internally. Files checked: `main.rs`, `svc.rs`, `a2a_client.rs`,
`transports/grpc.rs`, `transports/mesh.rs`, `shell/*.rs`, `audit.rs`,
`agent.rs`, `communication.rs`, `execution.rs`, `smoke.rs`, `registry.rs`.

**Root cause:** The function `client_tls()` returns a tonic 0.14 type (matching
`nexus-a2a`'s `tls::load_client_config_from_env()` return type), but references
it via bare `tonic::` which doesn't exist in `nexus-agent`'s namespace ‚Äî only
`tonic_14` (the Cargo alias) is available.

**Step 1.1.1** ‚Äî In `nexus-agent/src/a2a_client.rs`, change line 45:

```rust
// before
fn client_tls() -> Option<tonic::transport::ClientTlsConfig> {
// after
fn client_tls() -> Option<tonic_14::transport::ClientTlsConfig> {
```

**Verify:**
```bash
cargo check -p nexus-agent 2>&1 | grep -E "^error"
# Expected: no output

# Confirm no other bare tonic references remain:
grep -rn 'tonic::' nexus-agent/src/
# Expected: no output
```

## Stage 1.2 ‚Äî Fix latent Windows build.rs import

`nexus-agent/build.rs` has two compilation paths controlled by
`#[cfg(target_os = "windows")]` (line 19, compile-time gate on the function
definition) and `cfg!(target_os = "windows")` (line 11, runtime branch in
`main()`). The Windows-gated function `build_keylogger_bof()` uses
`Command::new("cl.exe")` at lines 33 and 46 but never imports
`std::process::Command`.

**Current imports in the file:**
- Top-level (lines 1‚Äì2): `std::env`, `std::path::PathBuf`
- Inside function (line 21): `std::fs`
- Missing: `std::process::Command`

**Why this doesn't fail today:** The `#[cfg(target_os = "windows")]` attribute
on line 19 excludes the entire function from compilation on Linux. The non-Windows
stub (line 100‚Äì103) calls `create_empty_bof_placeholder()` instead.

**Why it must be fixed:** Any Windows build (native or cross-compile via
`--target x86_64-pc-windows-gnu`) will fail with `cannot find value 'Command'`.

**Step 1.2.1** ‚Äî In `nexus-agent/build.rs`, add `use std::process::Command;`
after line 21 (`use std::fs;`), inside the Windows-gated function body:

```rust
#[cfg(target_os = "windows")]
fn build_keylogger_bof() {
    use std::fs;
    use std::process::Command;  // <-- added
    ...
```

**Verify:**

```bash
# Expected: line 22 inside the cfg(windows) function
```

## Stage 1.3 ‚Äî Remove invalid hickory-dns dependency

`nexus-infra/Cargo.toml:59` declares `hickory-dns = { workspace = true }`,
backed by `Cargo.toml:59` `hickory-dns = "0.24"` in workspace dependencies.

**Build warning:**
```
warning: nexus-infra v0.1.0 ignoring invalid dependency `hickory-dns` which is missing a lib target
```

**Why it's invalid:** `hickory-dns` 0.24 is a binary-only crate (DNS server CLI
tool) ‚Äî it has no `[lib]` target. Cargo resolves the dependency but cannot link
it as a library, so it emits a warning and ignores it.

**Scope:** Zero Rust code references `hickory_dns` anywhere in the workspace.
The DNS resolution code in `nexus-infra/src/domain_manager.rs:7` correctly uses
`hickory_resolver::{config::*, TokioAsyncResolver}`. No other crate in the
workspace depends on `hickory-dns`.

**Step 1.3.1** ‚Äî Delete the `hickory-dns` line from `nexus-infra/Cargo.toml`:

```toml
# before (lines 58-60)
# DNS and networking
hickory-dns = { workspace = true }
hickory-resolver = { workspace = true }

# after (lines 58-59)
# DNS and networking
hickory-resolver = { workspace = true }
```

**Step 1.3.2** ‚Äî Delete the `hickory-dns` line from the workspace root
`Cargo.toml` `[workspace.dependencies]` section:

```toml
# before (lines 58-60)
trust-dns-resolver = "0.23"
hickory-dns = "0.24"

# after (lines 58-59)
```

**Verify:**
